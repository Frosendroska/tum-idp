Retries       : not recorded (no error_retry_report.txt; error_analysis.png shows HTTP 200 only)

CONTEXT: This is a ramp-only experiment — NO steady_state phase. Three ramp phases step through
concurrency 768 → 1536 → 2304. This run uses different (smaller) object sizes than Run 1:
average object size is ~48.8 MB (9,069 GB / 185,724 requests) vs ~97.7 MB in Run 1. This
fundamentally changes the latency profile — latency at equivalent concurrency (768) is only
1,861 ms here vs 4,209 ms in Run 1. This run starts at the optimal concurrency level identified
in Run 1 (768), so no inverted-U shape is visible: throughput is monotonically declining as all
three tested concurrencies are at or above the saturation point. RTT stays remarkably flat
(163–251 ms across all phases), in stark contrast to Run 1's extreme RTT anomaly (up to 5,925 ms).

=== CROSS-PLOT FINDINGS & KEY OBSERVATIONS ===

1. MONOTONIC THROUGHPUT DECLINE: Unlike Run 1 (inverted-U peaking at 768), this run shows pure
   monotonic decline: 110.7 → 107.5 → 93.8 Gbps. This is because the experiment starts at
   concurrency 768, already at the optimal level. The sub-optimal low-concurrency regime (384 in
   Run 1) is never tested. This confirms that 768 is the minimum saturation concurrency for the
   c6in.32xlarge at this object size.

2. SMALLER OBJECT SIZE — DIFFERENT LATENCY REGIME: Average object size is ~48.8 MB (vs ~97.7 MB
   in Run 1). At 768 concurrency, mean latency is 1,861 ms (vs 4,209 ms in Run 1) — a 2.3× difference
   directly attributable to object size. This demonstrates that latency in these benchmarks is
   dominated by transfer time, not connection overhead.

3. NO RTT ANOMALY: RTT stays flat at 163–251 ms across all phases (vs 150–5,925 ms in Run 1).
   The absence of the dramatic RTT escalation at high concurrency seen in Run 1 may relate to the
   smaller object sizes completing faster, reducing R2-side queue pressure.

4. CONCURRENCY 768 CONTINUES TO SATURATE THE NIC: Peak throughput at ramp_1 (110.7 Gbps) exceeds
   the nominal 100 Gbps NIC rating, confirming ENA Express burst behaviour is consistent across
   object size configurations. The saturation point at 768 connections appears stable regardless
   of object size.

5. STEEP DECLINE FROM 1536→2304: The throughput drop from 1536 to 2304 connections (−14.5 Gbps,
   ~13.5%) is much larger than 768→1536 (−3.6 Gbps, ~3.3%). This accelerating decline suggests
   R2-side processing pressure increases super-linearly beyond ~1,536 connections.

6. VERY HIGH REQUEST THROUGHPUT: 246.5 req/s overall (vs 124.2 in Run 1) due to the smaller
   objects. At peak (ramp_1), 268.9 req/s — the highest request rate of any c6in experiment.

7. EFFICIENCY COLLAPSES AT HIGH CONCURRENCY: 768→0.141, 1536→0.068, 2304→0.039 Gbps/conn. The
   2304-connection efficiency (0.039 Gbps/conn) is the lowest of any concurrency tested across
   all c6in experiments. Each additional connection above 1536 yields negligible throughput.

8. PERFECT RELIABILITY: 100% success rate across 185,724 requests — the highest request count of
   any single run in this experiment series. The R2 service handles extreme concurrency (2304) and
   high request rates without any permanent failures.

9. END-OF-RUN DRAIN ARTEFACT: Both timeline plots show a sharp throughput collapse in the final
   30-second window of ramp_3, dropping to ~15 Gbps then ~0 Gbps. Consistent with all other runs.
