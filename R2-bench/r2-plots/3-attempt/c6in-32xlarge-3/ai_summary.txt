CONTEXT: This is a ramp-only experiment — NO steady_state phase. Three ramp phases step through
concurrency 768 → 1536 → 2304. Object size is ~48.8 MB average (8,044 GB / 164,757 requests),
identical to run 2 (~48.8 MB). Despite matching run 2's concurrency range AND object size, this
run produces a strikingly different throughput profile: an inverted-U peaking at 1536 (not
monotonic decline as in run 2). The key differentiator is that ramp_1 (768 concurrency) performs
much lower here (79 Gbps avg) than in run 2 (111 Gbps avg). The per-second timeline reveals
ramp_1 is NOT flat — it starts at ~63 Gbps and gradually climbs to ~100 Gbps over 248 seconds,
suggesting the pipeline needed significantly longer to saturate than in run 2. RTT stays flat
(138–229 ms), consistent with run 2 and unlike run 1's extreme RTT anomaly.

--- throughput_timeline.png ---
Line plot, 30-second prorated windows. X-axis = wall clock time (08 16:44–17:01). Y-axis = Gbps.
Colour-coded: red=warmup, blue=ramp_1, green=ramp_2, purple=ramp_3. No steady_state.

Observed behaviour:
  1. Warmup (16:44–16:45, red): starts ~72–75 Gbps, rises to ~79 Gbps then one data point near
     ~97 Gbps — high variance reflecting initial pipeline fill + early spike.
  2. Ramp_1 (16:45–16:49, blue): GRADUAL UPWARD TREND from ~77 Gbps to ~98 Gbps over the full
     248-second phase. Not the flat band seen in run 2 at the same concurrency. The phase
     average (79 Gbps) is dragged down by the slow start.
  3. Ramp_2 onset (~16:52, green): abrupt step-up to ~103 Gbps in first 30 s window, then holds
     very flat at ~105–108 Gbps throughout. Extremely stable.
  4. Ramp_3 (16:56–17:00, purple): maintains ~103–108 Gbps — nearly identical to ramp_2 in the
     30 s view. Sharp terminal drop to ~15 Gbps then ~0 Gbps (end-of-run drain artefact).

KEY FINDING: The most important feature is ramp_1's gradual upward trend (not flatness). By
the end of ramp_1 the throughput had nearly converged to ramp_2 levels (~98 vs ~105 Gbps),
suggesting that with a longer duration ramp_1 would have matched ramp_2. The inverted-U shape
in phase averages is therefore a ramp-duration artefact more than a true concurrency effect.

=== CROSS-PLOT FINDINGS & KEY OBSERVATIONS ===

1. INVERTED-U PEAKING AT 1536 — A THIRD DISTINCT PATTERN: Run 1 peaks at 768, run 2 declines
   monotonically from 768, and this run peaks at 1536. All three use the same instance and
   overlapping concurrency ranges, but differ in conditions. The inverted-U here is genuine
   but partially influenced by the slow ramp-up in ramp_1 artificially lowering the 768 average.

2. SLOW-START AT CONCURRENCY 768: ramp_1 takes 248 seconds to climb from ~63 Gbps to ~98 Gbps.
   By contrast, run 2's ramp_1 at the same concurrency was immediately at ~109 Gbps. The cause
   appears to be heavy extreme-latency outliers (20,000–60,000 ms) early in ramp_1 that drain
   pipeline capacity. As these outliers clear, throughput rises. This phenomenon was not present
   in runs 1 or 2 to this degree.

3. OBJECT SIZE MATCHES RUN 2 BUT THROUGHPUT DIFFERS: Both run 2 and run 3 have ~48.8 MB average
   object size, yet their throughput profiles are completely different (monotonic decline vs
   inverted-U). Object size alone does not explain the difference; run conditions (cache state,
   request distribution, object selection) are likely factors.

4. RAMP_2 AT 1536 IS THE PEAK (108.25 Gbps): This is comparable to run 1's ramp_2 peak
   (~113 Gbps at 768) and run 2's ramp_1 (~110.68 Gbps at 768). All three runs achieve ~108–113
   Gbps peak throughput, but at different concurrency levels, suggesting the c6in.32xlarge has a
   consistent R2-limited throughput ceiling around 108–113 Gbps regardless of concurrency path.

5. RAMP_3 AT 2304 DROPS BUT REMAINS >85 Gbps: The decline from 1536 to 2304 (−21.1 Gbps,
   ~19.5%) is larger than run 2's 768→1536 drop (−3.2 Gbps) but smaller than run 2's 1536→2304
   drop (−14.5 Gbps). The 2304-concurrency per-second band (~100–120 Gbps) was actually high,
   but the long ramp_3 duration (319.3 s) includes an end-of-run drain that lowers the average.

6. FLAT RTT — NO ANOMALY: RTT stays at 138–229 ms across all phases. Consistent with run 2
   and the absence of the dramatic RTT escalation seen in run 1 (up to 5,925 ms at 1536
   concurrency). Both run 2 and run 3 with smaller objects avoid the RTT anomaly.

7. HEAVY EXTREME OUTLIERS AT ALL CONCURRENCY LEVELS: All three concurrency groups have dense
   outlier clouds extending to ~59,000–60,000 ms, unlike other runs where outlier severity
   increased monotonically with concurrency. The dataset maximum (59,938 ms) belongs to
   concurrency 1536 — the best-performing phase in throughput terms.

8. SMALLEST REGRESSION SLOPE: y = 1.7x + 757.0 ms/connection — the shallowest latency-vs-
   concurrency relationship across all c6in.32xlarge runs (run 1: 6.1, run 2: 2.0). This
   reflects the fact that mean latency differences between concurrency levels are modest relative
   to within-level variance dominated by the heavy tails.

9. PERFECT RELIABILITY: 100% success rate across 164,757 requests (the second-highest request
   count of any c6in experiment, after run 2's 185,724). No errors at any concurrency level.

10. END-OF-RUN DRAIN ARTEFACT: Both timeline plots show a sharp throughput collapse to ~15 Gbps
    then ~0 Gbps in the final segment of ramp_3, consistent with all other c6in runs.
