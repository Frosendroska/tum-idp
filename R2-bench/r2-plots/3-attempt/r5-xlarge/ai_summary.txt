KEY FINDINGS AND INTERPRETATION
=============================================================================

1. THROUGHPUT CEILING AT ~6.5–6.7 Gbps REACHED AT 24 CONNECTIONS:
   The r5.xlarge hits a hard throughput ceiling around 6.5–6.7 Gbps at concurrency 24.
   Adding connections to 36 or 48 provides no additional throughput (+0.05 / -0.06 Gbps
   from the ramp_2 baseline). This ceiling likely reflects the instance's sustained NIC
   capacity (nominal 10 Gbps but effectively ~6.5 Gbps for large sustained transfers),
   or a per-endpoint R2 limit. This is qualitatively different from the inverted-U on
   larger instances — here throughput never declines, it simply saturates.

2. LOWEST THROUGHPUT OF ALL INSTANCES TESTED:
   6.06 Gbps average (6.65 Gbps at ceiling) vs. 44–104 Gbps on larger instances.
   The r5.xlarge is not suitable for high-throughput R2 workloads. It is a
   cost-efficiency optimized choice for low-bandwidth workloads only.

3. STEEPEST LATENCY-CONCURRENCY SLOPE (y=103.5x+207.3):
   103.5 ms per additional concurrent connection. Because throughput is capped, each
   new connection beyond 24 contributes only latency cost, no throughput benefit.

4. RTT ESCALATION CONFIRMS SATURATION:
   149.3 ms (conc 12) → 612.6 ms (conc 24) → 1,954.8 ms (conc 36) → 3,486.4 ms (conc 48).
   A 23× RTT increase from min to max concurrency confirms heavy queueing at the
   server or network layer beyond 24 connections. This follows the same pattern as
   hpc7g-16xlarge (also strong RTT escalation) but is more severe in absolute terms
   relative to the throughput achieved.

5. CLEANEST INTER-GROUP LATENCY SEPARATION:
   The four concurrency groups (12, 24, 36, 48) have nearly non-overlapping latency
   distributions. The minimum of the 48-connection group (2,544 ms) exceeds the
   median of the 12-connection group (1,538 ms). This textbook-clean separation makes
   the r5.xlarge the easiest instance to reason about: each concurrency level occupies
   a distinct, well-defined operating point.

6. LARGE OBJECT SIZE (~97.6 MB):
   Same as hpc7g-16xlarge and c6in-32xlarge run 1. Large objects amplify the per-request
   transfer time and make discrete request completions highly visible at 1-second
   throughput resolution (high per-second variance).

7. RAMP_1 SLOW-START AT 12 CONNECTIONS:
   Within ramp_1, throughput gradually rises from ~4.4 to ~5.6 Gbps over ~197 s.
   This is a TCP slow-start / connection establishment effect at very low concurrency:
   with only 12 HTTP connections, each connection's congestion window must grow
   independently before achieving line rate. The effect disappears at higher concurrency
   where multiple connections overlap their slow-start phases.

8. SMALLEST DATASET:
   6,156 requests — smallest of all instances tested. A consequence of low throughput
   (~6 Gbps) combined with large objects (~97.6 MB): only ~7 requests/second on average.
   Statistical confidence in tail percentiles is lower than for datasets with 36,000+
   requests (e.g., hpc7g-16xlarge: 36,072; c6in-32xlarge: 118,000+).
