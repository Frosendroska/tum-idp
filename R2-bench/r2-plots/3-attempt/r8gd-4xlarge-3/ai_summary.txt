=== CROSS-PLOT FINDINGS & KEY OBSERVATIONS ===

1. IMMEDIATE THROTTLING ONSET: The most critical difference from Run 2 is that throttling
   engaged within the first 2 minutes of steady state (vs ≈28 min into steady state in Run 2).
   Both runs converge to the same ≈7.1 Gbps floor. This pattern strongly suggests a burst-
   credit or token-bucket mechanism: Run 2 exhausted its burst credits after ≈28 min; Run 3
   started with nearly depleted credits (due to the prior Run 2 occurring only ≈73 min earlier),
   so the burst allocation was exhausted almost immediately.

2. IDENTICAL THROTTLED FLOOR: Both Run 2 and Run 3 converge to precisely ≈7.1–7.2 Gbps in the
   throttled regime. The near-identical floor value (within 1%) across two independent runs
   separated by >1 hour confirms this is a hard infrastructure-imposed ceiling, not a
   measurement artefact.

3. RAMP PHASES FULLY UNAFFECTED: Both runs achieve ≈14 Gbps across all ramp phases (warmup,
   ramp_1, ramp_2, ramp_3). The short duration (<4 min) of each ramp segment does not deplete
   burst credits. This suggests the burst threshold is volume-based (~900 GB or ≈28 min at
   14 Gbps), not concurrency-based.

4. HIGHER OVERALL LATENCY THAN RUN 2: The overall mean (6,424 vs 5,602 ms), P95 (13,982 vs
   11,347 ms), and P99 (18,148 vs 14,957 ms) are all higher in Run 3. This is a direct
   consequence of spending nearly 100% of steady-state time in the throttled regime (vs ≈53%
   in Run 2). In the throttled regime, R2 delivers data at half the speed, roughly doubling
   transfer latency for the same object size.

5. FEWER TOTAL REQUESTS: Run 3 completed 45,864 requests vs Run 2's 58,287 (≈21% fewer).
   At 8.97 Gbps average (vs 11.40 Gbps), Run 3 transferred 4,479 GB vs 5,692 GB. Both runs
   ran for the same wall-clock duration (1.19 hours), so the throughput difference directly
   explains the request count gap.

6. PERFECT RELIABILITY ACROSS BOTH RUNS: Despite significant throughput throttling, R2 never
   returned errors or permanently failed requests in either run. The service degrades gracefully
   under sustained high load by reducing bandwidth, not by dropping connections.

7. CONCURRENCY-96 IS THE STRESS POINT: In both runs, concurrency-96 (the steady-state level)
   bears the full weight of throttled performance. Concurrency-48 and 144 (short ramp phases)
   always see the high-throughput regime. An experiment design that kept concurrency at 48 or
   144 for the full steady state would likely delay or avoid throttling onset, since each
   connection contributes less total volume per unit time at 48 connections than at 96.
