Concurrency   : 96 (warmup/ramp_1) → 192 (ramp_2) → 288 (ramp_3) → 192 (steady_state)
Retries       : 58 requests needed 1 retry (0.048% retry rate); max retry_count=1;
                total HTTP round-trips=120,094; error_retry_report.txt present

CONTEXT: This is a ramp + steady_state experiment. Four short ramp phases step through
concurrency 96 → 192 → 288 (~11 minutes total), then a long steady_state phase runs at
concurrency 192 for ~60.7 minutes. The defining feature of this run is a dramatic
burst-credit throttling cliff occurring at approximately 07 23:28–23:30 (~37 minutes into
steady_state), where throughput collapses from ~14.3 Gbps to ~7.1 Gbps (a ~50% reduction)
and remains at the lower level for the remainder of the run. This is the same burst-credit
depletion mechanism seen in r8gd.4xlarge runs 1 and 2. The run uses HTTP 206 range-reads
(not HTTP 200 full-object GET as in c6in/c5n experiments).

=== CROSS-PLOT FINDINGS & KEY OBSERVATIONS ===

1. BURST-CREDIT THROTTLING CLIFF AT ~23:30: The central finding of this run. After ~37 minutes
   of steady_state at 192 concurrency, throughput halves from ~14.3 Gbps to ~7.1 Gbps in a
   single 30-second measurement window. This is consistent with EBS or network burst credit
   exhaustion on the r8gd.4xlarge instance. The cliff is permanent: throughput does not
   recover for the remaining ~25 minutes of the run.

2. CONCURRENCY-INSENSITIVE BURST MODE: During burst mode, all ramp phases deliver nearly
   identical throughput regardless of concurrency: 96→14.21, 192→14.30, 288→14.30 Gbps.
   Unlike c6in instances where concurrency strongly affects throughput, the r8gd.4xlarge
   appears to be limited by a burst credit budget rather than by connection count.

3. V-SHAPE ARTIFACT IN THROUGHPUT VS CONCURRENCY: The apparent V-shape (96=14.2, 192=10.9,
   288=13.6 Gbps) is entirely caused by the 192-concurrency group averaging pre-cliff burst
   and post-cliff throttled data. It does NOT reflect a real relationship where concurrency
   192 performs worse than 96 or 288. This is an important interpretation caveat.

4. POST-CLIFF THROUGHPUT ~7 Gbps IS STABLE: After the cliff, throughput holds at ~7.0–7.3 Gbps
   with very small variance (30 s windows). This is the baseline sustained NIC bandwidth of
   the r8gd.4xlarge without burst credits.

5. FLAT RTT THROUGHOUT — NO RTT ANOMALY: RTT values are 103–151 ms across all ramp phases
   and 109 ms in steady_state. The dramatic RTT escalation seen in c6in-32xlarge run 1
   (up to 5,925 ms) is completely absent here. This may reflect the shorter object sizes in
   range-read requests or different R2-side queuing behaviour for this instance type.

6. DOMINANT STEADY_STATE SAMPLE: 87% of all requests (104,397/120,036) are at concurrency 192
   in steady_state. This means all aggregate statistics (overall mean latency, P95, P99) are
   heavily weighted by steady_state behaviour. The 192-concurrency mean of 4,659.7 ms blends
   burst-phase latency (~3,000–4,000 ms) with throttled-phase latency (~6,000–8,000 ms).

7. EXTREME OUTLIER AT CONCURRENCY 96 (52,856 ms): The single highest-latency request at
   concurrency 96 (the lowest level) is unusual. It appears in the latency_over_time scatter
   near 22:40 (very start of warmup), strongly suggesting a pipeline initialization stall
   where a single request was queued before the pipeline was fully active.

8. MINIMAL RETRIES (58 TOTAL): Only 0.048% of requests needed a single retry. Max retry_count=1.
   The retries are not concentrated at the throttling cliff, indicating the cliff causes
   throughput degradation but not request failures.

9. LONGEST RUN IN THIS EXPERIMENT SERIES: At 1.19 hours (4,289 seconds), this is substantially
   longer than the ramp-only c6in/c5n runs (~11–26 minutes). The long steady_state (3,644 s)
   is what allows the burst credit exhaustion to be observed. Shorter runs would show only
   the burst regime.

10. HTTP 206 vs HTTP 200: This run uses range-reads (HTTP 206), unlike c6in and c5n experiments
    which use full-object GET (HTTP 200). Latency and throughput comparisons across instance
    types must account for this difference in request type and object access pattern.
